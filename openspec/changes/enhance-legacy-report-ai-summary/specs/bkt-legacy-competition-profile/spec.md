## MODIFIED Requirements

### Requirement: Legacy student navigation exposes four first-level modules
The legacy student product SHALL expose four first-level modules labeled `主页`, `学习`, `评测`, and `报告`.

#### Scenario: Legacy student opens the old product
- **WHEN** an authenticated student opens `web-student-old`
- **THEN** the first-level navigation MUST show exactly the student modules `主页`, `学习`, `评测`, and `报告`
- **AND** `主页` MUST be the default active module
- **AND** the bottom navigation active tab treatment MUST remain square with no rounded modern-pill styling
- **AND** the navigation MUST NOT show Atom, RAG, Agent, learning assistant, intelligent monitoring, provider, retrieval-diagnostic, `TKE`, or `TKT` entries

#### Scenario: Student switches modules
- **WHEN** the student selects a first-level module
- **THEN** the old app MUST route to that module without loading the modern student app shell
- **AND** selecting `报告` MUST open the old learning-report surface rather than a generic personal-center page
- **AND** current `web-student` routes and navigation MUST remain unchanged

## ADDED Requirements

### Requirement: Legacy report module presents reports instead of a personal center
The old student `报告` module SHALL present completed learning and assessment reports as its primary purpose, with student identity shown only as supporting context.

#### Scenario: Student opens legacy reports
- **WHEN** an authenticated student opens the old `报告` module
- **THEN** the page MUST show a student identity card with student id, name, and class when available
- **AND** the page MUST show report-oriented heading copy such as `学习报告`, `测评报告`, or `BKT 学情报告`
- **AND** the page MUST render an old-style switch between `概况` and `历史报告`
- **AND** `概况` MUST be the default view and MUST show report count, average score, pending wrong-question review count, and the latest assessment/report summary when available
- **AND** the latest assessment/report summary MUST provide a `查看报告` action that opens that report detail directly
- **AND** the page MUST NOT present itself primarily as `我的`, `个人中心`, account settings, or a profile-management surface

#### Scenario: Student opens historical reports
- **WHEN** the student selects `历史报告`
- **THEN** the page MUST render the student's available report records as list or table-like old-style cards
- **AND** the list MUST show at most 10 reports per page
- **AND** pagination controls MUST show the current page, total page count, and previous/next page actions

#### Scenario: Report list is empty
- **WHEN** the student has no report records available
- **THEN** the old report page MUST render a controlled old-style empty state explaining that reports appear after completing assessment or learning cycles
- **AND** the empty state MAY offer a navigation action to `评测`
- **AND** it MUST NOT show raw backend diagnostics, SQL, provider names, TKE/TKT labels, or retrieval/model wording

#### Scenario: Report list is inspected
- **WHEN** visible report-list copy is inspected
- **THEN** report rows MUST summarize student-facing facts such as report title, completion time, score, answered count, wrong count, and review status when available
- **AND** report rows MUST NOT expose raw TKE/TKT values, raw mastery probability, raw BKT state vectors, JSON payloads, provider names, Agent/RAG terms, chunk ids, embedding ids, or retrieval diagnostics

### Requirement: Legacy report detail provides AI learning summaries without exposing TKE internals
The old student product SHALL provide a report detail view with a learning summary generated by the configured backend AI report capability, translated into plain student-facing conclusions without exposing implementation internals.

#### Scenario: Student opens a report detail
- **WHEN** the student selects a report from the old report list or opens a just-completed assessment result report
- **THEN** the old app MUST open an old-style report detail view
- **AND** the detail view MUST show a concise score or completion overview when available
- **AND** the detail view MUST show an `AI 学情总结`, `AI 学习总结`, or equivalent legacy-facing summary section
- **AND** the summary MUST describe what the student did well, which experiments or point types need review, and what to do next
- **AND** the summary MUST be understandable without reading TKE/TKT/BKT formulas or raw mastery numbers

#### Scenario: Internal mastery evidence exists
- **WHEN** the report payload includes mastery changes, point mastery evidence, weak point coverage, wrong answers, or next recommendations
- **THEN** the old report generation path MUST pass that evidence to the configured AI report capability to produce natural-language conclusions
- **AND** the summary MUST NOT display raw `mastery_score`, `mastery_prob`, TKE, TKT, probability vectors, draw tickets, or low-level algorithm fields to the student
- **AND** the summary MAY use legacy-facing terms such as `掌握度`, `薄弱点`, `复盘`, `巩固`, `学情`, and `智能总结`

#### Scenario: AI generation is unavailable
- **WHEN** AI summary generation is unavailable, times out, or returns an unsafe/internal response
- **THEN** the old report detail MUST render a deterministic legacy-facing fallback summary based on score, wrong count, covered experiments, and available recommendations
- **AND** the fallback MUST NOT mention provider errors, Agent errors, RAG errors, model names, API keys, OpenAI, Qwen, BGE, chunk, embedding, or retrieval internals
- **AND** the report detail MUST remain usable for the student

#### Scenario: Current product is inspected
- **WHEN** current `web-student` assessment reports are inspected after this change
- **THEN** current report behavior MUST remain unchanged unless a separate current-product change explicitly modifies it
- **AND** old report AI summary code MUST remain scoped to old frontend or old/legacy backend adapters when mainline APIs are not a safe fit

### Requirement: Legacy report detail provides wrong-question explanations
The old student report detail SHALL provide wrong-question review using the AI-generated mistake explanation for the completed report when available.

#### Scenario: Report has wrong answers
- **WHEN** a report contains wrong answers
- **THEN** the report detail MUST render a wrong-question section
- **AND** each wrong-question item MUST show the question stem, option list when available, `做错项`, `正确选项`, and an `AI 解析` block
- **AND** the `AI 解析` block MAY use the stored question-bank/database explanation when no structured per-question AI explanation exists
- **AND** the UI MUST NOT render a single global `AI 错题解析` paragraph that crowds out the per-question cards
- **AND** the answer comparison MUST be compact and MUST NOT use oversized answer boxes that crowd the explanation

#### Scenario: AI mistake explanation exists
- **WHEN** the persisted report has `mistake_explanation.source = ai`
- **THEN** the old report detail MAY keep that text in the report payload for backend provenance
- **AND** the old visible UI MUST still prioritize per-question review cards over a global mistake-explanation paragraph
- **AND** the UI MUST NOT expose the underlying provider, model, Agent mode, RAG retrieval, prompt, or diagnostic trace

#### Scenario: AI mistake explanation is unavailable
- **WHEN** AI mistake explanation generation is unavailable, times out, or returns unsafe/internal text
- **THEN** the old UI MUST render stored question explanations or deterministic fallback explanations that still tell the student to compare their answer with the reference answer and revisit the related experiment point
- **AND** no visible error state may expose provider, model, Agent, RAG, prompt, retrieval, SQL, or JSON diagnostics

#### Scenario: Report has no wrong answers
- **WHEN** a report contains no wrong answers
- **THEN** the report detail MUST render a positive old-style state such as `本次没有错题`
- **AND** it MAY suggest continuing to the next experiment or a harder assessment
- **AND** it MUST NOT render empty wrong-question chrome or internal payload placeholders

### Requirement: Legacy report generation preserves old product boundaries
Legacy report generation and display SHALL reuse current assessment/report identities while keeping old-specific AI summary and explanation behavior isolated from mainline product semantics.

#### Scenario: Legacy report backend support is needed
- **WHEN** existing student assessment-report endpoints cannot provide the old report detail payload required by `web-student-old`
- **THEN** the implementation MUST add old-scoped backend routes, schemas, or services under a legacy namespace
- **AND** those routes MAY read current `student_assessment_reports`, smart assessment sessions, question attempts, question-bank explanations, and assessment payload snapshots
- **AND** those routes MUST NOT require old-only question records, old-only experiment records, old-only mastery rows, old-only report ids, or a separate legacy database

#### Scenario: Legacy assessment is submitted
- **WHEN** the old student submits a smart, custom, random, all-range, or point-scoped assessment
- **THEN** scoring and BKT/mastery updates MUST still use current backend assessment identities and mechanisms
- **AND** the normal report creation path MUST call the configured current AI assessment-report generator rather than a legacy-only local template
- **AND** report creation or AI explanation generation MUST NOT make the answer submission hang indefinitely
- **AND** if AI report hydration is slow, the old product MUST be able to show the score result first and hydrate the report detail later or on demand

#### Scenario: Legacy report AI output is inspected
- **WHEN** generated or fallback report text is rendered in old student UI
- **THEN** it MAY use visible legacy-facing terms such as `AI总结`, `AI学情总结`, `AI错题解析`, `掌握度`, `薄弱点`, `复盘建议`, `参考答案`, and `解析`
- **AND** it MUST NOT show visible terms such as `Agent`, `RAG`, `Atom`, `OpenAI`, `Qwen`, `BGE`, `chunk`, `embedding`, `rerank`, `prompt`, `retrieval`, `向量`, `ES诊断`, `TKE`, or `TKT`

#### Scenario: Legacy report tests are run
- **WHEN** maintainers validate old report behavior
- **THEN** tests MUST cover the `报告` navigation label, report list rendering, report detail rendering, no visible TKE/TKT/raw mastery values, wrong-question explanation fallback, and forbidden internal terms
- **AND** backend tests MUST cover any old-scoped report endpoint route registration, auth boundary, and use of shared report/question identities
