## Context

### Current state in the codebase

Student H5 currently has an experiment point/detail page, but the page is still thin. `apps/student-web/src/features/experiments/ExperimentDetailPanel.tsx` renders a video stage, a compact experiment summary card, a list of candidate points, a disabled practice strip, optional AI action, and finish-learning action. The API contract `StudentExperimentDetailResponse` only contains experiment summary fields, `video_candidates`, and `videos`. It does not expose teacher-authored principle, phenomenon explanation, safety note, or editable related point links.

Admin experiment management already exposes a point-like video binding workflow. `apps/admin-web/src/features/experiments/ExperimentsPage.tsx` loads `/api/admin/experiments/{experiment_id}/video-points`, renders point cards, and lets teachers bind existing videos to a point. The backend currently constructs these points from two sources:

- `formal_experiments.metadata.video_candidates`, where point keys are generated by hashing the candidate title and index.
- `media_bindings.metadata.point_key` and `media_bindings.metadata.point_title`, where uploaded or bound videos keep point metadata.

This means point identity is not yet a first-class database object. If candidate titles are edited or regenerated, derived point keys can drift. If a media binding has a point key that is not in the candidates list, the admin endpoint treats it as a stored point, but there is no single point record owning display order, title, publication status, or learning content.

Backend AI/RAG already has a different point-related system: `experiment_video_point_evidence` stores manually reviewed point evidence bindings keyed by `(experiment_id, point_key)`, with `experiment_chunk_ids`, `theory_chunk_ids`, manual review status, and review grade. `scripts/import_manual_reviewed_point_evidence.py` imports those bindings from `data/seed/point_evidence/manual_reviewed_point_evidence.jsonl`. The assistant runtime hydrates these chunk ids into a fixed point evidence package for point-context questions. This system is valuable and remains protected, but it is not a student page content source.

Student video-library search already has a route and backend adapter boundary from `student-h5-video-library-search-entry`. The current implementation builds in-memory `VideoLibraryDocument` objects from published experiments, media bindings, candidate titles, chapter/profile metadata, and some metadata fields. The ES adapter only searches an existing index by id. There is no authoritative index builder, no required ES container, no IK analyzer mapping, and no point content facts to index beyond metadata.

### Product correction from exploration

The new requirement is not "generate text from AI and show it under the video." The correct product model is:

```text
Teacher-authored point content
  -> PostgreSQL fact source
    -> student experiment point/detail display
    -> Elasticsearch/IK search projection
      -> student video-library search

AI point chunk/evidence bindings
  -> manually reviewed evidence package
    -> assistant/RAG/question diagnostics
```

Both systems can share `(experiment_id, point_key)` as a point identity, but they have different authors, storage, publication gates, consumers, and risk boundaries.

### Search architecture research absorbed into this design

The attached research proposes a PostgreSQL plus Elasticsearch CQRS shape, with chemical dictionary support, IK analyzers, formula normalization, equation parsing, reaction feature extraction, and domain stopwords. The useful parts should be adopted, but adapted to this repository:

- The source table is not a generic `experiments` table. The index unit should be an experiment video point or point learning content document.
- PostgreSQL remains the fact source. ES is a derived read model for student video-library search.
- Chemical formula parsing should happen before indexing, not by feeding equations raw to a Chinese tokenizer.
- IK should handle Chinese terms, while backend preprocessing should normalize Unicode subscripts/superscripts, formula aliases, reagent synonyms, and reaction symbols.
- Search results must remain typed and actionable, routing to second-level/detail pages.

## Goals / Non-Goals

**Goals:**

- Make experiment video points first-class stable database records.
- Store point learning content as human-authored, teacher/admin edited, published content.
- Keep video resources bound to points without making media assets own the learning copy.
- Add admin UI for point content editing inside experiment management.
- Refactor the student H5 experiment point/detail page to show:
  - video
  - experiment principle
  - phenomenon explanation
  - safety note
  - related experiment links
  - go-test action
- Default related links to nearby points in the same parent experiment while allowing manual edit or override.
- Keep "go test" as a fixed action into the existing assessment flow, not editable content.
- Make ES with IK analyzer support a required production-like application service for video-library search.
- Build and synchronize ES documents from published, student-visible point content.
- Preserve local deterministic search fallback only as a development/test escape hatch where explicitly configured.
- Keep AI point chunk/evidence bindings separate from published student content.

**Non-Goals:**

- Do not generate student page principle, explanation, or safety copy from AI automatically.
- Do not remove or rewrite `experiment_video_point_evidence`, `source_chunks`, or manual-reviewed point evidence imports.
- Do not make AI/RAG evidence the source of student-facing point page body text.
- Do not turn the video library into all-site/global search.
- Do not require video transcripts/ASR in this change.
- Do not make the test entry manually editable per point.
- Do not replace the existing objective question bank, post-learning assessment flow, or point-aware question bank metadata.
- Do not move point content editing into the generic video asset library page.

## Decisions

### Decision 1: Store stable experiment video points as first-class records

Add an `experiment_video_points` table keyed by `(experiment_id, point_key)`. The point key is server-controlled and stable. The title can be edited without changing the key. Seed/backfill should initialize rows from existing `formal_experiments.metadata.video_candidates` and media binding point metadata.

Suggested shape:

```text
experiment_video_points
- experiment_id text references formal_experiments(id)
- point_key text
- point_title text
- display_order int
- source text: seed_candidate | media_binding | manual
- status text: active | archived
- metadata jsonb
- created_at / updated_at
- primary key (experiment_id, point_key)
```

Rationale: many systems already depend on point keys: video bindings, question-bank point metadata, assistant point evidence, analytics, and route targets. A first-class table turns that implicit contract into a durable one.

Alternative considered: continue deriving point keys from candidate title hashes. Rejected because title edits would risk key drift and because point content needs an owner row even when no video is bound yet.

### Decision 2: Store learning copy separately from video media

Add a point learning content table keyed by `(experiment_id, point_key)`. Media assets remain reusable binary/video resources. Content belongs to the point because a point may have multiple videos, a video may be replaced, and the principle/explanation/safety note should not disappear when a media binding changes.

Suggested shape:

```text
experiment_point_learning_content
- experiment_id text
- point_key text
- principle_mode text: equation | text
- principle_equation text null
- principle_text text null
- phenomenon_explanation text
- safety_note text
- content_status text: draft | published | archived
- published_at timestamptz null
- published_by uuid null
- created_by uuid null
- updated_by uuid null
- metadata jsonb
- created_at / updated_at
- primary key (experiment_id, point_key)
```

Validation:

- `principle_mode = equation` requires `principle_equation` and forbids using `principle_text` as the primary principle.
- `principle_mode = text` requires `principle_text` and forbids using `principle_equation` as the primary principle.
- Published content requires phenomenon explanation and safety note.
- Published content must have an active point and a published parent experiment.
- Videos can be absent, but the student page must render a graceful video empty state.

Rationale: this keeps teacher-authored display copy explicit and auditable.

Alternative considered: store content in `formal_experiments.metadata` or `media_bindings.metadata`. Rejected because metadata JSON would hide validation, make publishing and audit weak, and blur ownership between experiment, point, and media.

### Decision 3: Keep AI evidence as a separate consumer path

The existing AI point evidence system remains as-is in principle:

```text
experiment_video_point_evidence
  -> source_chunks
  -> assistant fixed point evidence package
  -> AI/RAG answer grounding and diagnostics
```

It can share the stable point identity from `experiment_video_points`, and implementation may validate that evidence rows resolve to active points. It must not populate student body copy fields. It may be shown to admins as reference/context if a future design explicitly adds that, but not as an auto-published source.

Rationale: AI-generated or AI-selected chunk bindings have a different consumer and trust model. The student page is a curated content surface. The assistant evidence package is a RAG grounding surface.

Alternative considered: use manual-reviewed evidence chunks to auto-fill principle/explanation/safety. Rejected because the user explicitly wants these fields edited by humans, and because evidence chunks often cite broad textbook sections rather than final student-facing copy.

### Decision 4: Put point editing inside experiment management

The admin frontend should extend the existing experiment detail drawer/workspace. The point card should evolve from "point video reference" into a point workspace with sections:

- point title and status
- video bindings and publish/unpublish controls
- experiment principle editor
- phenomenon explanation editor
- safety note editor
- related links editor
- publish state and validation summary
- search index status or last sync state

The generic video resource page remains asset-centric: upload, processing, duplication, preview, file lifecycle. It should not become the place where teachers author point content.

Rationale: teachers think in experiments and points when authoring learning copy. Video files are assets, not learning content containers.

Alternative considered: add content fields to video resources. Rejected because a point can have multiple videos and the same point content should survive video replacement.

### Decision 5: Related links use default nearby points plus manual curation

Related links are editable, but the system should provide useful defaults. When no manual links exist, the student point page should default to nearby points in the same parent experiment or same concrete experiment group. Teachers can add other point links manually and can hide or reorder defaults where supported.

Suggested shape:

```text
experiment_point_related_links
- source_experiment_id
- source_point_key
- target_experiment_id
- target_point_key
- relation_type text: default | manual
- sort_order int
- hidden boolean
- created_by / updated_by
- created_at / updated_at
```

Implementation can either persist default links during migration or derive them at read time and overlay manual rows. The design preference is derived defaults plus persisted manual overrides, so backfill does not need to materialize every adjacency row.

Rationale: same-experiment neighboring points are the safest default, but teachers need to connect cross-experiment learning when chemistry warrants it.

Alternative considered: make links purely manual. Rejected because it creates too much authoring work before the student page is useful.

### Decision 6: The student point page renders from published content, not search documents

Student detail APIs should fetch from PostgreSQL and return structured point content. ES is not used to render the point page. ES is only used for search. If search navigates to a point, the detail page still fetches the canonical published point content from the backend.

Student page layout:

```text
Page bar
Video stage
Point summary/context
Experiment principle
Phenomenon explanation
Safety note
Related experiment links
Go test
Finish learning / existing completion behavior
```

The page should still record learning events and preserve assistant/feedback context. If AI action remains available, its context can include the human-authored point summary plus point key, but the assistant still uses its own evidence package when answering.

Rationale: search indexes are optimized for retrieval and highlighting, not as the authoritative display record.

Alternative considered: render the ES hit source directly after search. Rejected because stale or partial index documents could produce inconsistent student page content.

### Decision 7: Keep "go test" as a fixed assessment handoff

The "go test" action is not an editable point field. It should use the existing post-learning/assessment flow for the point's experiment/chapter context, matching current student H5 behavior. Related point content may improve context, but it must not create a new per-point manually authored test destination.

Rationale: testing belongs to the question bank and assessment flow. Teachers should not have to maintain a button per point.

Alternative considered: store a per-point test URL or test label in point content. Rejected as unnecessary and likely to drift from the existing objective question bank.

### Decision 8: Build the video-library index from published point learning content

The search document unit should be a published, student-visible experiment point. Videos are attached as availability and action context. Experiments may still appear as grouped or secondary result types, but the primary document should be point-centered.

Suggested document shape:

```json
{
  "id": "video_point:EXP_19_3_04:candidate-1-122909e7",
  "document_type": "video_point",
  "experiment_id": "EXP_19_3_04",
  "experiment_code": "19-3-04",
  "experiment_title": "硫代硫酸盐的性质",
  "parent_code": "19-3",
  "parent_title": "实验 19-3 硫及其化合物",
  "chapter_ids": ["CH14"],
  "point_key": "candidate-1-122909e7",
  "point_title": "Na2S2O3 + HCl",
  "principle_mode": "equation",
  "principle_equation": "Na2S2O3 + 2 HCl = 2 NaCl + S↓ + SO2↑ + H2O",
  "principle_text": "",
  "phenomenon_explanation": "teacher-authored text",
  "safety_note": "teacher-authored text",
  "formula_chemicals": ["na2s2o3", "hcl", "nacl", "s", "so2", "h2o"],
  "chemical_aliases": ["硫代硫酸钠", "盐酸", "二氧化硫"],
  "reaction_features": ["沉淀", "沉淀反应", "气体", "气体生成"],
  "reagents": ["Na2S2O3", "HCl"],
  "phenomenon_tags": ["浑浊", "硫沉淀", "刺激性气体"],
  "video_count": 1,
  "has_published_video": true,
  "route_target": {
    "kind": "point_detail",
    "experiment_id": "EXP_19_3_04",
    "point_key": "candidate-1-122909e7"
  },
  "updated_at": "..."
}
```

Rationale: the user expects search results to route to second-level pages. Point documents naturally route to a point detail page.

Alternative considered: index generic experiment records only. Rejected because students search for point-level phenomena and expect direct navigable results.

### Decision 9: Elasticsearch with IK is required for application deployment

Production-like deployments should include an Elasticsearch-compatible service with IK analyzer support. Docker Compose and deployment docs should define the service, health check, index bootstrap command or startup job, environment variables, and validation.

Local/test behavior:

- Unit tests can keep deterministic local fallback for fast isolated tests.
- Development may allow local fallback only when explicitly configured.
- Production readiness should require ES/IK configuration and a healthy index.

Rationale: the user explicitly wants ES as an application necessary part. Search quality for Chinese chemistry terms and formulas cannot be treated as optional after this change.

Alternative considered: keep ES optional with local fallback as the default. Rejected because it would hide search quality issues until late deployment.

### Decision 10: Use IK for Chinese text and backend preprocessing for chemistry

The ES mapping should use IK for Chinese text fields and whitespace/keyword-style fields for extracted chemistry tokens.

Index-time preprocessing should include:

- Unicode chemistry normalization:
  - `₂ -> 2`, `₃ -> 3`, `₄ -> 4`, superscript charge forms to normalized tokens where practical.
- Formula cleanup:
  - strip stoichiometric coefficients when extracting chemicals
  - strip state markers like `(aq)`, `(s)`, `(g)`, `(l)`, `浓`, `稀`, `溶液`
  - preserve the original equation for display/search snippets
- Reaction feature extraction:
  - `↓` means `沉淀`, `沉淀反应`
  - `↑` means `气体`, `气体生成`
  - `Δ`, `加热`, or `微热` means heat-related tags
  - common color/phase words should become tags where deterministically extractable
- Chemical dictionary and aliases:
  - examples: `HCl`/`盐酸`, `H2O2`/`过氧化氢`/`双氧水`, `Na2S2O3`/`硫代硫酸钠`
  - aliases should be maintained in backend data/config and used in search expansion/indexing
- Stopwords:
  - generic words such as `实验`, `反应`, `现象`, `生成`, `产生`, `观察`, `加入`, `滴加` should have reduced search impact.

Search should query weighted fields, for example:

- `formula_chemicals^6`
- `chemical_aliases^6`
- `reaction_features^5`
- `point_title^4`
- `principle_equation^3`
- `phenomenon_explanation^3`
- `safety_note^2`
- `experiment_title^2`
- `chapter/family/property labels`

Rationale: IK is useful for Chinese text but not sufficient for chemical formulas. The formula/equation path needs deterministic backend structure.

Alternative considered: rely entirely on IK tokenization over raw text. Rejected because equations and formula symbols are easily broken by generic tokenizers.

### Decision 11: Synchronize index updates through an explicit index projection service

Admin saves should write PostgreSQL first. Publishing or unpublishing point content should enqueue or trigger an index update for the affected point. A reliable design is:

```text
admin save/publish
  -> PostgreSQL transaction
  -> search indexing outbox row
  -> index worker/upsert
  -> index status updated
```

For small local deployments, the implementation may perform immediate upsert after commit plus a durable retry marker, but it should still provide a rebuild endpoint/command to regenerate the entire index from PostgreSQL.

Index deletion rules:

- Draft content is not indexed.
- Archived point content is deleted from the index.
- Archived/draft parent experiments are deleted from the index.
- Unpublished/unready videos do not expose playable media in the document.
- A point can be indexed without a published video only if product decides browse/search should surface content-only points; otherwise require published content plus at least one playable video. The current recommendation is to index published content and expose `has_published_video` so the frontend can label availability.

Rationale: direct ES writes inside request handling are fragile. A projection/outbox model makes search updates observable and recoverable.

Alternative considered: rebuild the whole index after every edit. Rejected because it scales poorly and makes admin edits feel unpredictable.

### Decision 12: Preserve typed route targets in video-library responses

The existing video-library result contract should remain actionable. Result items should include stable route targets with `experiment_id`, `point_key`, `point_title`, optional `profile_id`, `chapter_id`, and source context. A search hit without a resolvable student route should be omitted or rendered unavailable.

Rationale: the video library is a learning launcher, not a passive search engine result page.

Alternative considered: return raw ES hits and let the frontend infer URLs. Rejected because route construction must remain centralized and safe.

### Decision 13: Backfill must preserve current point keys

Migration should backfill `experiment_video_points` from current data:

1. For each existing `formal_experiments.metadata.video_candidates`, generate the current candidate key using the same `_candidate_point_key(index, title)` algorithm.
2. Insert active point rows with display order and title.
3. Scan `media_bindings` for experiment bindings with point metadata.
4. Insert any stored media point keys not already present, preserving key/title.
5. Keep `media_bindings.metadata.point_key` and `point_title` for compatibility, but treat `experiment_video_points` as canonical going forward.
6. Validate that question bank point references and `experiment_video_point_evidence` rows resolve to canonical point rows.

Rationale: the current question bank, evidence, analytics, and video bindings already depend on existing point keys.

Alternative considered: generate new point UUIDs and map old keys to new ids. Rejected because it would ripple through question banks, evidence, routes, and analytics.

### Decision 14: Student display should degrade explicitly when content is missing

During migration, many point rows may not yet have published point content. Student APIs should not invent content. The frontend should render a controlled unavailable state for missing unpublished content while still allowing existing video playback where appropriate, depending on the chosen rollout gate.

Recommended rollout gate:

- Existing point detail pages can still open during migration.
- New structured sections appear only when content is published.
- The video-library ES index includes only published point content.
- Admin validation surfaces missing content before publication.

Rationale: this allows incremental authoring without pretending generated content exists.

Alternative considered: auto-fill missing sections from experiment summary or AI evidence. Rejected because it violates the human-authored source rule.

## Risks / Trade-offs

- [Risk] Point key drift breaks question/evidence/search links. -> Mitigation: backfill with the existing key algorithm, make point key server-controlled, add validation that all media bindings, question metadata, and evidence rows resolve.
- [Risk] Teachers may publish incomplete or unsafe point content. -> Mitigation: require principle mode, principle body, phenomenon explanation, safety note, and validation before publish; show preview before publication.
- [Risk] ES becomes stale after admin edits. -> Mitigation: use outbox/index status, retries, rebuild command, and tests for publish/update/unpublish/delete flows.
- [Risk] ES/IK deployment adds operational weight. -> Mitigation: define Docker service, analyzer bootstrap, health checks, env vars, readiness validation, and fallback rules clearly.
- [Risk] IK tokenization still mishandles formulas. -> Mitigation: extract formula chemicals, aliases, and reaction features in backend and index them in dedicated fields.
- [Risk] Search becomes too broad. -> Mitigation: index only experiment point learning content and student-visible video resources; reject unrelated domains.
- [Risk] Existing AI point evidence is mistaken for display content. -> Mitigation: document and test the two consumer paths; do not join `experiment_video_point_evidence` into student point content APIs.
- [Risk] Manual content authoring takes time. -> Mitigation: allow drafts, show admin completeness filters, keep existing point pages operational during migration, and index only published content.
- [Risk] Related-link defaults produce noisy navigation. -> Mitigation: default only to nearby same-parent points and allow manual hide/reorder/add.
- [Risk] Chemical aliases become stale or incomplete. -> Mitigation: keep a maintainable dictionary/config and add tests for important synonyms such as `HCl`/`盐酸`, `H2O2`/`双氧水`, and `Na2S2O3`/`硫代硫酸钠`.

## Migration Plan

1. Add database migration for `experiment_video_points`, `experiment_point_learning_content`, related links, and search index/outbox/status tables if needed.
2. Backfill point rows from current `formal_experiments.metadata.video_candidates` and experiment media binding metadata.
3. Validate all current media bindings, question point references, and manual-reviewed evidence rows resolve to canonical points.
4. Add backend services for point listing, point content editing, validation, publication, related link derivation/manual overrides, and student detail payloads.
5. Add ES/IK service to Docker/deployment configuration and document required environment variables.
6. Add index bootstrap/mapping creation with IK analyzers, chemistry char filters, synonym support, and stopword support.
7. Add backend index projection builder and sync/rebuild commands.
8. Extend admin experiment detail UI from video point binding to point content editing and publishing.
9. Extend student point/detail API and frontend page to render the new structured content.
10. Update video-library search API to query ES point documents and preserve typed route targets.
11. Add tests for publication visibility, no AI evidence leakage, search indexing, ES fallback rules, and mobile rendering.
12. Run full validation: OpenSpec strict validation, backend tests, frontend typecheck/tests/build, student mobile QA, production resource validation, and ES readiness checks.

Rollback strategy:

- Disable the new admin point content editor if necessary while preserving the migrated point rows.
- Keep student point detail on the previous summary/video fallback if structured content is not available.
- Rebuild or clear the ES index from PostgreSQL; ES is a derived projection.
- Do not delete or rewrite `experiment_video_point_evidence` or source chunk data during rollback.

## Open Questions

- Should a point with published learning content but no published video appear in video-library search with a "video unavailable" label, or should video-library search require at least one published playable video?
- Should related default links be derived at read time only, or materialized in the database for auditability?
- Should the chemical dictionary be a database table editable by admins, a seed/config file maintained by developers, or both?
- Should index updates be synchronous after publish for the first implementation, or should the outbox worker be implemented immediately?
- Should the student point detail page keep a contextual AI action in this layout, and if yes where should it sit relative to the fixed content and go-test action?
